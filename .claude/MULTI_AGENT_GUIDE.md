# 多 Agent 操作手册 - vid_tokenizer 研究项目

管理多个 Claude Agent 协作进行视频压缩研究的规范。

## 四种核心角色（研究版本）

### 1. Planner（研究规划者）

**职责：**
- 分析研究问题和已有工作
- 设计实验方案和对比策略
- 规划实验序列（从简到复杂）
- 评估 baseline 和竞品

**何时使用：**
- 规划新的实验方向
- 设计 baseline 复现方案
- 制定 tokenizer 对比计划
- 评估研究进度和后续方向

**输出：**
```
- 实验设计方案（假设、配置、指标）
- Baseline 列表和复现计划
- Tokenizer 对比矩阵
- 实验时间表
```

---

### 2. Coder（模型实现者）

**职责：**
- 实现压缩模型和 tokenizer 集成
- 编写训练和评估代码
- 创建实验配置和脚本
- 进行单元测试和快速验证

**何时使用：**
- 实现新的模型变种
- 集成新的 tokenizer
- 编写数据加载和预处理
- 修复代码 bug

**输出：**
```
- 训练脚本和配置文件
- 数据处理代码
- 单元测试
- 验证报告
```

---

### 3. Researcher（研究分析者）

**职责：**
- 分析实验结果和性能指标
- 对比与 baseline 的差异
- 进行学术文献调查
- 提供研究方向建议

**何时使用：**
- 分析实验结果
- 理解为什么某方法更好
- 研究相关的论文和方法
- 评估技术方案

**输出：**
```
- 结果分析报告
- 文献综述
- 技术方案对比
- 研究方向建议
```

---

### 4. Verifier（实验验证者）

**职责：**
- 验证实验结果的有效性
- 检查是否满足可重复性标准
- 对比不同实验的一致性
- 识别异常或有问题的结果

**何时使用：**
- 完整训练后的结果验证
- Baseline 复现的正确性检查
- 实验数据质量评估
- 论文准备前的最终检查

**输出：**
```
- 验证报告
- 异常检测
- 可重复性评估
- 批准或需改进清单
```

## Agent 协作模式（研究工作流）

### 模式 1: Baseline 复现流程

```
Planner
  ├─ 选择要复现的 baseline
  ├─ 设计复现方案
  └─ 生成配置方案
       ↓
    Coder
      ├─ 实现模型和配置
      └─ 运行训练
           ↓
        Verifier
          ├─ 检查结果与论文的对比
          ├─ 验证数据有效性
          └─ 生成验证报告
               ↓
            [成功?]
            ├─ 是 → 记录为可信 baseline
            └─ 否 → 反馈给 Coder 调查差异
```

### 模式 2: 新想法验证流程

```
Researcher
  ├─ 调查相关文献
  ├─ 分析可行性
  └─ 提出技术方案
       ↓
    Planner
      ├─ 设计实验对比
      ├─ 定义成功指标
      └─ 创建配置
           ↓
        Coder
          ├─ 实现方法变种
          ├─ 创建实验脚本
          └─ 运行全训练
               ↓
            Researcher
              ├─ 分析结果
              ├─ 与 baseline 对比
              └─ 评估改进
                   ↓
                [值得继续?]
                ├─ 是 → 规划下一步
                └─ 否 → 记录学到的东西
```

### 模式 3: 并行 Tokenizer 测试

```
Planner
  ├─ 选择 3 个候选 tokenizer
  ├─ 设计标准化配置
  └─ 分配给 3 个 Coder
       ↓
    Coder-A (CLIP)  ┐
    Coder-B (ViT)   ├─ 并行运行
    Coder-C (自监督) ┘
       ↓
    Verifier
      ├─ 对比 3 个结果
      ├─ 检查一致性
      └─ 生成对比表
           ↓
    Researcher
      └─ 分析哪个 tokenizer 最优
```

## Agent 通信与信息交接

```markdown
## 交接给 [下一个 Agent]

### 关键信息

- 约束：[约束]
- 已完成：[已做的]

### 需要做的
- [ ] 任务1

### 上下文
- 相关文件：[位置]
- 配置文件：[位置]
- 参考资料：[资源]
```

**Coder → Researcher**
```
- 训练日志位置
- 模型检查点位置
- 配置文件
- 完整的性能指标
```

**Researcher → Planner**
```
- 性能对比表
- 关键发现
- 建议方向
- 所需资源评估
```

## 最佳实践

### ✅ 好的多 Agent 协作

- **清晰的接力棒** - 每个 Agent 知道自己的责任和下一步
- **充分的上下文** - 包含所有必要的背景信息
- **定期同步** - 不是埋头工作然后才发现问题
- **明确的约束** - 时间、资源、成功指标清晰

### ❌ 糟糕的多 Agent 协作

- **信息遗漏** - Agent 猜测上下文
- **无序并行** - 多个 Agent 做重复工作
- **缺乏同步** - 发现问题时已经浪费时间
- **移动目标** - 需求不断改变

## 故障排除

### 问题：Agent 产生了不兼容的配置

**解决：**
```
Planner 明确定义标准配置
  ↓
所有 Coder 使用相同的模板
  ↓
Verifier 检查一致性
```

### 问题：结果出乎预料，无法解释

**解决：**
```
1. Verifier 检查数据有效性
2. Coder 检查代码实现
3. Researcher 查看相关论文寻找解释
4. Planner 设计后续实验来验证
```

### 问题：某个实验 Agent 卡住了

**解决：**
```
立即通知其他 Agent → 改变方法或寻求帮助
不要等待，影响其他实验的进度
```

## 与单 Agent 工作的对比

| 方面 | 单 Agent | 多 Agent |
|------|---------|---------|
| 速度 | 顺序（慢） | 并行（快） |
| 简单性 | 更简单 | 需要协调 |
| 错误检测 | 较晚 | 较早（验证环节） |
| 资源利用 | 低 | 高 |
| 适用 | 简单任务 | 复杂研究项目 |

**何时用多 Agent：**
- 有多个独立的实验可同时进行
- 需要专家级的代码审查和结果分析
- 项目复杂度高，需要不同的专长

**何时用单 Agent：**
- 实验有强依赖关系（必须串行）
- 项目规模小（< 1 周）
- 工作需要深度探索而非并行执行
