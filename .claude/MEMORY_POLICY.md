# 记忆机制与策略 - vid_tokenizer 研究项目

Claude 持久化记忆系统在视频压缩研究中的应用规范。

## 记忆系统概览

### 三层记忆架构

```
┌──────────────────────────────────┐
│   会话记忆                        │ 这次实验的细节、临时结论
│   （不会保留到下次）              │
├──────────────────────────────────┤
│   项目记忆 (CLAUDE.md)            │ 研究问题、baseline、约束
│   (本项目内持久化)                │
├──────────────────────────────────┤
│   团队记忆 (.claude/memory/)      │ 跨实验的经验、陷阱、最佳实践
│   (项目级别持久化)                │
└──────────────────────────────────┘
```

## 团队记忆（.claude/projects/vid_tokenizer/memory/）

### 四种记忆类型（研究特定）

#### 1. User 类型 - 研究者信息

**何时记录：**
- 研究兴趣和专长
- 偏好的方法论
- 已知的知识缺口
- 工作风格和偏好

**示例：**
```markdown
---
name: 用户研究背景
type: user
---

用户专注于机器理解的视频表示学习。
偏好严谨的实验设计和可重复的结果。
对 INR 方法感兴趣但新手。
```

#### 2. Feedback 类型 - 研究规范和陷阱

**何时记录：**
- 用户纠正的研究方法
- 被验证的最佳实践
- 团队中踩过的坑
- 实验规范和约束

**示例：**
```markdown
---
name: 实验数据泄漏陷阱
type: feedback
---

实验中发现训练数据无意中混入了验证集。

**Why:** 测试/验证分割脚本有 off-by-one 错误。

**How to apply:**
- 总是手动检查数据划分
- 在 experiments.md 中明确记录数据分割 ID
- 运行验证：两个集合应该没有重叠的视频 ID
```

#### 3. Project 类型 - 项目进度和背景

**何时记录：**
- 当前的 baseline 性能
- 已尝试和失败的方向
- 研究里程碑
- 已解决的主要问题

**示例：**
```markdown
---
name: Baseline 性能记录
type: project
---

当前最优 baseline：简单 INR + CLIP tokenizer
- 压缩率: 100 倍
- L_tok: 0.045 (↓0.15 from original)
- 下游任务准确率: 68.2% (vs 71.5% baseline)

**Why:** 这是我们能复现的最稳定结果。

**How to apply:** 新实验应该以此为基准，报告与其相比的改进。
```

#### 4. Reference 类型 - 外部资源

**何时记录：**
- 关键数据集位置
- 预训练模型位置
- Wandb 项目链接
- GPU/计算资源

**示例：**
```markdown
---
name: 数据集和资源位置
type: reference
---

- Egocentric 数据集: /data/egocentric_videos/
- CLIP 模型权重: ~/.cache/huggingface/
- Wandb 项目: https://wandb.ai/user/vid-tokenizer
- 结果输出: ./results/
```

## 会话记忆管理

### 实验进行中

**记录的内容：**
- 当前实验的关键参数
- 中间结果和观察
- 临时假设和待验证的想法
- 发现的问题

**不记录到持久化记忆中的内容：**
- 训练日志细节
- 中间 checkpoint 的性能
- 临时的调试信息

### 实验完成后

**更新持久化记忆：**
1. 是否发现了新的陷阱？ → feedback_XXX.md
2. 项目状态是否改变？ → project_video_compression.md
3. 学到了什么关键经验？ → user_research_insights.md

## 记忆使用指南

### 何时读取记忆

**任务开始时：**
- 了解项目当前状态（project_video_compression.md）
- 查看已知陷阱（所有 feedback_*.md）
- 确认 baseline 性能（CLAUDE.md）

**面临相似决策时：**
- 选择 tokenizer？ → 查看项目记忆中的对比
- 数据处理？ → 查看 feedback_data_*.md
- 遇到训练问题？ → 查看 feedback_training_*.md

### 何时忽视记忆

✅ **信任记忆：**
- 记忆来自已完成的实验
- 有明确的日期和上下文

❌ **不信任记忆：**
- 用户明确说"忽视记忆"
- 记忆与当前代码矛盾（相信代码）
- 发现记忆过时（立即更新/删除）

## 记忆生命周期

### 创建

```
实验发现 → 用户确认 → 写入记忆文件 → 更新 MEMORY.md 索引
```

**文件格式示例：**
```markdown
---
name: 发现名称
type: feedback / project / reference / user
---

[核心内容，1-2段]

**Why:** [背景和原因]

**How to apply:** [如何在未来应用这个知识]
```

### 维护

- **定期审计** - 每月检查过时的记忆
- **冲突解决** - 发现重复或矛盾时合并
- **更新标记** - 记录记忆最后验证日期

### 清理

**删除的情况：**
- 记忆已被证实是错误的
- 发现了更准确的替代方案
- 项目方向改变，使其失效

### Feedback 记忆示例

```markdown
---
name: 随机种子固定陷阱
type: feedback
---

多次实验结果不可重复的原因：PyTorch 的随机数生成器未固定。

**Why:** 小的随机变化会导致大的结果差异，特别是在收敛边界上。

**How to apply:**
```python
torch.manual_seed(42)
np.random.seed(42)
# 在每个实验开始时设置
```
```

## 最佳实践

### ✅ 好的记忆

- 清晰、具体、可操作
- 包含背景和原因
- 有明确的适用范围
- 易于搜索和理解

### ❌ 糟糕的记忆

- 模糊或假设性的描述
- 缺乏背景信息
- 重复或与其他记忆冲突
- 难以找到或理解
