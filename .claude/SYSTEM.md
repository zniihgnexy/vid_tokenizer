# 系统哲学 - vid_tokenizer 项目

Claude Code 工作流在视频压缩研究项目中的设计原则与价值观。

## 核心价值观

### 1. 机器理解优先（Machine-First）
- 优化目标是 **下游机器任务的可用性**，不是人眼视觉质量
- 所有压缩决策都应考虑：tokenizer 一致性是否保留
- 记住：`y_hat ≈ y` 是核心目标，而不是 PSNR

### 2. 科学严谨性
- 实验必须可重复：固定随机种子、清晰的配置、完整的日志
- 每个实验有明确的假设和验证指标
- 对比必须严格：baseline 一致、单变量改变

### 3. 渐进式验证
- 从简单的 baseline 开始（INR + 简单 tokenizer）
- 逐步增加复杂度：多 tokenizer、3D 信息等
- 每个阶段都有量化的性能指标

### 4. 设计约束意识
- 记住项目的 **固定核心**（见 CLAUDE_readme.md）
- 不要默认假设普遍适用性（INR 需要每视频拟合）
- 区分：per-video fitting vs method-level generalization

## 研究工作流哲学

### 三阶段模式

1. **规划** (Plan)
   - 明确实验假设
   - 设计 baseline 和对比
   - 确认 tokenizer 选择

2. **执行** (Execute)
   - 实现和训练
   - 监控 tokenizer 一致性损失
   - 记录完整的实验配置

3. **分析** (Analyze)
   - 对比机器理解性能（不只是 PSNR）
   - 分析 tokenizer 表示是否保留
   - 决定下一步方向

## 关键术语映射

| 符号 | 含义 | 备注 |
|------|------|------|
| `x` | 原始视频 | 输入 |
| `z` | 压缩表示 | 紧凑、已量化 |
| `x_hat` | 重建视频 | 解压后的结果 |
| `y = T(x)` | 原始视频的机器表示 | tokenizer 输出（特征） |
| `y_hat = T(x_hat)` | 重建视频的机器表示 | 应与 y 接近 |
| `T` | tokenizer | 特征提取，不是文本 tokenizer |
| `L_tok` | tokenizer 一致性损失 | 核心目标：`\|\|y_hat - y\|\|^2` |

## 错误处理

### 研究特定的错误分类

1. **数据问题** - 数据泄漏、分割错误、质量问题
   - 恢复：检查数据预处理、重新划分、验证标签

2. **实验问题** - 随机种子不一致、配置遗漏
   - 恢复：固定随机种子、明确所有参数、复现

3. **性能问题** - tokenizer 一致性没有改进、压缩太高
   - 恢复：可视化 y vs y_hat、调整损失权重、检查 tokenizer 选择

4. **方法问题** - baseline 复现失败、核心目标不清晰
   - 恢复：回到标准 baseline、重读 CLAUDE_readme.md、获取指导
