# 错误处理与恢复机制 - vid_tokenizer 研究项目

研究特定的错误分类、恢复流程和预防策略。

## 研究特定的错误分类

### 1. 数据问题

#### 数据泄漏
**症状：** 验证集性能异常好，不符合预期

**恢复：**
```
1. 检查数据分割逻辑
   - 有重复的视频 ID 吗？
   - 训练和验证集有重叠吗？
2. 重新生成数据分割
3. 重新训练，验证性能是否合理
4. 更新记忆中的数据泄漏陷阱
```

#### 数据不足或不均衡
**症状：** 某类视频的性能特别差

**恢复：**
```
1. 分析数据分布
   - 每类有多少样本？
   - 类别间差异大吗？
2. 考虑数据增强或重采样
3. 更新 CLAUDE.md 中的数据描述
```

### 2. 随机性问题

#### 不可重复的结果
**症状：** 同一配置运行多次，结果不同

**恢复：**
```
1. 检查所有随机数生成器
   - torch.manual_seed(42) 是否设置？
   - np.random.seed(42) 是否设置？
   - CUDA_DETERMINISTIC 环境变量？
2. 检查是否有非确定性操作（某些 GPU 算子）
3. 将随机种子记录到 experiments.md
4. 运行 3 次并报告均值±标差
```

**代码片段：**
```python
def set_seed(seed=42):
    torch.manual_seed(seed)
    np.random.seed(seed)
    torch.cuda.manual_seed_all(seed)
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False
```

### 3. Tokenizer 一致性问题

#### L_tok 没有下降
**症状：** tokenizer 一致性损失不改进

**诊断：**
```
1. 检查 y_hat 是否真的接近 y
   - 可视化特征空间
   - 计算 cosine distance
   - 检查是否有 NaN
2. 检查 tokenizer 是否被冻结
   - T(x) 应该固定（冻结）
   - 只有压缩模块学习
3. 检查损失权重
   - λ_tok 是否太小？
   - 是否被其他损失淹没？
```

**修复：**
```python
# 确保 tokenizer 被冻结
for param in tokenizer.parameters():
    param.requires_grad = False

# 检查 L_tok 的权重
if lambda_tok < 0.1:
    print("Warning: λ_tok too small, may not train effectively")
```

#### 重建视频没有保留足够的语义
**症状：** 重建视频可视上很好，但 tokenizer 表示差异大

**恢复：**
```
1. 确认这不是 tokenizer 限制
   - 用不同的 tokenizer 再试一次
2. 增加 λ_tok 权重
3. 检查压缩率是否太高
   - 减少压缩（增加 bitrate）
4. 考虑混合损失：L_tok + 感知损失
```

### 4. 实现问题

#### Baseline 无法复现
**症状：** 按照论文运行，结果低 10% 以上

**诊断：**
```
1. 检查配置
   - 所有超参是否正确？
   - 数据预处理是否一致？
   - 模型大小是否相同？
2. 检查代码
   - 有无 bug 或不同的实现选择？
   - 优化器设置是否相同？
3. 检查环境
   - PyTorch 版本不同？
   - GPU 影响（不同 GPU 可能有差异）？
```

**允许的差异：** ±2%（由于随机性和实现细节）

**超过差异的恢复：**
```
1. 运行 3 次，确认均值
2. 检查论文中是否有遗漏的细节
3. 查看原始代码（如公开）
4. 记录差异到 experiments.md，标记为已知问题
```

### 5. 训练问题

#### 训练不收敛或内存溢出
**症状：** 损失震荡或 OOM 错误

**恢复：**
```
1. 减小 batch size
2. 减少模型大小
3. 使用梯度累积替代大 batch
4. 检查是否有内存泄漏（循环中积累张量）
```

#### 模式崩溃（Mode Collapse）
**症状：** 重建所有帧都是平均值，特征相同

**恢复：**
```
1. 增加 L_rec 权重（迫使重建有差异）
2. 检查 tokenizer 梯度流是否正确
3. 使用多个 tokenizer 增加多样性约束
```

## 错误恢复流程

### 轻微错误（Typos、小参数错误）

**恢复时间：** 15分钟

```
1. 识别错误位置
2. 修改代码
3. 运行快速验证（部分数据）
4. 提交修复
```

### 中等错误（Logic bugs、配置问题）

**恢复时间：** 1-4 小时

```
1. 停止当前训练
2. 写入测试来重现问题
3. 定位根本原因
4. 实现修复
5. 完整验证（完整数据集）
6. 更新记忆以防止再发生
```

### 严重错误（方法错误、数据问题）

**恢复时间：** 半天到数天

```
1. 立即停止关联实验
2. 通知用户问题发现
3. 隔离问题：
   - 数据问题？重新处理
   - 方法问题？重新设计
   - 实现问题？大重构
4. 设计修复和验证计划
5. 彻底测试
6. 记录到 MISTAKE_POLICY 作为经验教训
```

## 常见问题库

### Q: 我的 L_tok 在减少，但压缩不够

**A:** 这是权衡问题。调整 λ_tok vs λ_rate 的比例。
```python
# 尝试降低 tokenizer 权重，允许更高压缩
λ_tok = 0.1  # from 1.0
λ_rate = 10.0  # from 1.0
```

### Q: Baseline 复现失败了怎么办？

**A:** ±2% 的差异是正常的。如果更大：
1. 检查论文中的所有细节
2. 查看原始代码（如公开）
3. 运行 3 次获得均值
4. 标记为"论文差异：原因未知"并继续

### Q: 我怀疑有数据泄漏

**A:** 立即检查：
```python
# 手动验证无重叠
train_ids = set(train_data['video_id'])
val_ids = set(val_data['video_id'])
assert len(train_ids & val_ids) == 0, "Data leak detected!"
```

### Q: 重建质量很高，但 L_tok 不改进

**A:** 这说明高压缩率和 tokenizer 一致性有冲突。选择：
1. 降低压缩率（增加保留的信息）
2. 改变 tokenizer（也许选定的 tokenizer 不适合高压缩）
3. 混合目标（L_tok + 感知损失）
