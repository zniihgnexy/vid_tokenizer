# 固定流程规范 - vid_tokenizer 研究项目

标准化的研究工作流，针对机器理解的视频压缩项目。

## 日常研究工作流

### 实验周期

```
1. 假设 → 2. 配置 → 3. 训练 → 4. 分析 → 5. 记录 → 6. 决策
```

### 1. 假设清晰化 (30分钟)

**检查清单：**
- [ ] 明确实验目标（改进什么？）
- [ ] 定义假设（为什么应该改进？）
- [ ] 确认对比基准（与什么对比？）
- [ ] 选定 tokenizer（用哪个特征提取器？）
- [ ] 设定成功指标（什么算成功？）

**记录位置：** 在 experiments.md 中创建实验条目

### 2. 配置设计 (30分钟)

**必须明确的：**
- [ ] 模型架构（INR 类型、参数）
- [ ] 压缩率目标（多少倍？）
- [ ] Tokenizer 选择（CLIP / 补丁 / 隐状态？）
- [ ] 损失权重（λ_rate, λ_rec, λ_tok）
- [ ] 训练超参（学习率、epoch、batch size）
- [ ] 数据集和分割（train/val/test）
- [ ] 随机种子（固定以保证可重复）

**创建配置文件：** `configs/exp_XXX.yaml`

### 3. 训练执行 (数小时)

**代码运行：**
```bash
# 激活环境
conda activate vid_tokenizer

# 运行训练
python -m src.train --config configs/exp_XXX.yaml

# 监控日志
tail -f logs/exp_XXX.log
```

**监控要点：**
- [ ] tokenizer 一致性损失（`L_tok`）是否下降？
- [ ] 重建质量（`L_rec`）和压缩率（`L_rate`）的平衡？
- [ ] GPU 内存是否溢出？
- [ ] 训练是否收敛？

### 4. 结果分析 (1小时)

**量化评估：**
- [ ] 机器理解性能指标（下游任务准确率）
- [ ] tokenizer 特征相似度（cosine distance）
- [ ] 压缩率（原始 vs 压缩文件大小）
- [ ] 重建 PSNR（仅供参考，不是主要指标）

**定性观察：**
- [ ] 重建视频质量如何？
- [ ] tokenizer 表示是否保留？（可视化特征）
- [ ] 任何意外发现？

### 5. 结果记录 (30分钟)

**更新 experiments.md：**
```markdown
### EXP-XXX: [实验名称]
**状态：** ✓ 完成
**日期：** YYYY-MM-DD

**关键结果：**
- 压缩率: X 倍
- L_tok: X.XX (vs baseline Y.YY)
- 下游任务精度: Z.Z% (vs baseline W.W%)

**配置：**
- Tokenizer: CLIP
- λ_tok: 1.0

**关键发现：**
1. [发现1]
2. [发现2]

**下一步：** [建议方向]
```

### 6. 方向决策 (30分钟)

**评估选项：**
- [ ] baseline 复现成功？
- [ ] 是否超越 baseline？
- [ ] 改进来自哪个方面（架构/损失/tokenizer）？
- [ ] 有新的有趣发现吗？

**更新记忆：**
- 如果发现新的陷阱，更新 feedback_experiment_standards.md
- 如果核心问题改变，更新 project_video_compression.md

## 多 Agent 协作工作流

### 规划 Agent → 实现 Agent → 验证 Agent

```
Planner 设计实验
  ├─ 分析研究问题
  ├─ 设计 baseline 和对比
  └─ 生成实验计划
       ↓
    Coder 实现
      ├─ 编写配置文件
      ├─ 实现模型
      └─ 运行训练
           ↓
        Verifier 验证
          ├─ 检查结果
          ├─ 对比 baseline
          └─ 发现问题
               ↓
            [问题？] ← 反馈给 Coder
             │
             ↓
           发布结果
```

## 特殊工作流

### Baseline 复现工作流

```
1. 获取原始论文或代码
   ↓
2. 设置复现配置 (configs/baseline_XXX.yaml)
   ↓
3. 运行训练，记录完整日志
   ↓
4. 对比报告指标 (允许 ±2% 差异)
   ↓
5. 成功？
   是 → 记录为可信 baseline
   否 → 调查差异原因、更新记忆
```

### Tokenizer 对比工作流

```
1. 选择 2-3 个候选 tokenizer（CLIP、ViT、自监督等）
   ↓
2. 使用相同的压缩配置运行
   ↓
3. 对比 L_tok 和下游任务性能
   ↓
4. 分析：哪个 tokenizer 特征最被保留？
   ↓
5. 结论：选择最优 tokenizer 或计划多 tokenizer 方法
```

### Bug 修复工作流

```
1. 问题症状清晰化
   - L_tok 没有下降？
   - 重建质量奇差？
   - 随机种子不可重复？
   ↓
2. 最小复现案例
   - 使用最小数据集
   - 禁用不必要的特性
   ↓
3. 根因分析
   - 数据泄漏？
   - 配置错误？
   - 实现 bug？
   ↓
4. 修复和验证
   - 修改代码
   - 运行快速测试
   - 重新训练确认
   ↓
5. 防止复发
   - 添加单元测试
   - 记录到陷阱清单
```

## 协作与沟通

### 同步点

1. **实验规划** - Planner 展示计划，用户批准
2. **配置完成** - 展示 config 文件，确认无问题
3. **首轮结果** - 展示初始实验结果，讨论方向
4. **最终报告** - 展示完整的实验总结和论文框架

### 信息隐藏原则

**避免过度沟通：**
- 不逐行解释训练代码
- 不重复已知的信息

- 为什么选择这个 tokenizer
- 实验结果与预期的差异
- 遗留的技术问题
- 后续的研究方向
